    • What you'll be able to do after completing the path

    New to FraiseQL? Start Here¶

    Goal: Understand what FraiseQL is and how it differs from traditional GraphQL servers.

    Total Time: ~45 minutes

    Path¶

    1. README.md (5 minutes)
    2. High-level overview
    3. Core principles
    4. Feature list

    5. Key Takeaway: FraiseQL is a compiled database execution layer, not a GraphQL server

    6. docs/prd/prd.md — Read Sections 1-2 only (20 minutes)

    7. Vision and philosophy
    8. System architecture

    9. Key Takeaway: Compilation over interpretation, database as source of truth

    10. docs/glossary.md — Skim, bookmark for reference (10 minutes)

    11. Key terminology

    12. Key Takeaway: AuthoringIR, CompiledSchema, database targeting

    13. docs/architecture/database/database-targeting.md — Read Sections 1-3 (10 minutes)

    14. Multi-database support via compile-time specialization
    15. Key Takeaway: Same schema source, different compiled outputs per database

    After this path, you'll understand:

    • What FraiseQL is and isn't
    • Core architectural principles
    • How multi-database support works
    • Key terminology

    Next Steps:

    • Choose a role-specific path below based on your work
    • Or continue to "Complete Documentation Path" for comprehensive coverage

    For Architects¶

    Goal: Understand FraiseQL's architecture, design decisions, and trade-offs.

    Total Time: ~3.5 hours

    Foundation (1 hour)¶

    1. README.md (5 min)
    2. docs/prd/prd.md (30 min)
    3. All sections
    4. Key Takeaway: Design principles, architectural boundaries
    5. docs/glossary.md (10 min)
    6. docs/architecture/database/database-targeting.md (15 min)
    7. Key Takeaway: Compile-time schema specialization

    Core Architecture (1.5 hours)¶

    1. docs/architecture/core/authoring-languages.md (20 min)
    2. Language-agnostic compilation via AuthoringIR

    3. Key Takeaway: One canonical language per org, translation paths for migration

    4. docs/architecture/core/compilation-pipeline.md (30 min)

    5. 7 phases from authoring → CompiledSchema

    6. Key Takeaway: Deterministic, static analysis at compile time

    7. docs/architecture/core/execution-model.md (40 min)

    8. 6 runtime phases

    9. Key Takeaway: Runtime is deterministic, executes compiled plans

    10. docs/architecture/integration/federation.md (40 min)

    11. Federation v2 architecture and implementation
    12. Direct DB federation for multi-database scenarios
    13. Key Takeaway: Subgraph composition via direct database connections + HTTP fallback

    Specifications (1 hour)¶

    1. docs/specs/compiled-schema.md (25 min)
    2. CompiledSchema JSON structure

    3. Key Takeaway: Immutable artifact, database-target-specific

    4. docs/specs/schema-conventions.md (35 min)

    5. Database patterns (tb_, v_, fn_*)

    6. Key Takeaway: Conventions enable automatic CQRS, efficient composition

    After this path, you'll be able to:

    • Explain FraiseQL's architecture to stakeholders
    • Understand compile-time vs runtime responsibilities
    • Evaluate FraiseQL for your use case
    • Design schema conventions for your domain

    Recommended Next:

    • docs/adrs/adr-009-federation-architecture.md — Federation design decisions

    For Analytics Engineers¶

    Goal: Build analytical systems with FraiseQL's fact table patterns and aggregations.

    Total Time: ~3-4 hours

    Core Concepts (1 hour)¶

    1. docs/architecture/core/compilation-vs-runtime.md (20 min)
    2. Understand compile-time vs runtime separation

    3. Key Takeaway: Python/TS authoring → Rust compilation → Rust runtime

    4. docs/architecture/core/execution-model.md (30 min)

    5. GraphQL query execution phases
    6. Phase 2.5: Aggregation Resolution

    7. Key Takeaway: How GROUP BY and aggregates are compiled

    8. docs/prd/prd.md — Section 3.5 only (10 min)

    9. Analytical Execution Semantics overview
    10. Key Takeaway: No joins principle, fact table patterns

    Analytics Architecture (1.5 hours)¶

    1. docs/architecture/analytics/fact-dimension-pattern.md ⭐ START HERE (40 min)
    2. Fact table structure (measures, dimensions, filters)
    3. No joins principle
    4. Aggregate tables = fact tables with different granularity

    5. Key Takeaway: All analytical tables use same pattern: SQL columns (measures) + JSONB (dimensions)

    6. docs/architecture/analytics/aggregation-model.md (30 min)

    7. GROUP BY compilation strategy
    8. Aggregate function selection
    9. Temporal bucketing (DATE_TRUNC, DATE_FORMAT, strftime, DATEPART)
    10. HAVING clause validation

    11. Key Takeaway: Compile-time schema analysis → optimized SQL

    12. docs/specs/analytical-schema-conventions.md (20 min)

    13. Naming conventions: tf_(fact tables, any granularity), td_ (dimension)
    14. Column patterns: measures, dimensions, denormalized filters
    15. Index recommendations
    16. Key Takeaway: Conventions required by FraiseQL compiler

    Database-Specific Implementation (1 hour)¶

    1. docs/specs/aggregation-operators.md (30 min)
    2. PostgreSQL: Full support (STDDEV, VARIANCE, FILTER)
    3. MySQL: Basic support (no statistical functions)
    4. SQLite: Minimal support
    5. SQL Server: Enterprise support (STDEV, VAR, JSON_VALUE)

    6. Key Takeaway: Database capability manifest determines available functions

    7. docs/architecture/database/database-targeting.md (15 min)

    8. Compile-time database specialization

    9. Key Takeaway: GraphQL schema matches database capabilities

    10. docs/specs/capability-manifest.md — Section 3.4 only (15 min)

    11. Aggregation operators in capability manifest
    12. Key Takeaway: How compiler knows which aggregates to generate

    Practical Application (30 min)¶

    1. docs/guides/analytics-patterns.md ⭐ PRACTICAL EXAMPLES (30 min)
      • 10 common query patterns with SQL execution
      • Simple aggregation, GROUP BY, temporal bucketing, HAVING
      • Performance optimization tips
      • Database-specific notes
      • Key Takeaway: Copy-paste patterns for real analytics queries

    After this path, you'll be able to:

    • Design fact tables with measures + dimensions
    • Write GraphQL aggregate queries (GROUP BY, HAVING, temporal bucketing)
    • Understand performance characteristics (SQL columns 10-100x faster than JSONB)
    • Use pre-aggregated fact tables (e.g., tf_sales_daily) for common rollups
    • Leverage database-specific aggregate functions

    Recommended Next:

    • docs/architecture/database/arrow-plane.md — Section 5.5 for BI tool integration
    • docs/architecture/analytics/window-functions.md — Window functions (ROW_NUMBER, LAG/LEAD)
    • docs/specs/window-operators.md — Window function reference
    • docs/specs/schema-conventions.md — Section 4.3.1 for analytical fact tables

    Important Notes:

    • FraiseQL does NOT support joins; all dimensions must be denormalized at ETL time
    • ETL is managed by DBA/data team; FraiseQL provides GraphQL query interface only
    • Pre-aggregated fact tables use same structure, just different granularity (e.g., tf_sales_daily vs tf_sales)

    Tips for Effective Reading¶

    1. Start with Your Role Path¶

    Don't try to read everything at once. Follow the path for your role first.

    2. Keep glossary.md Open¶

    Reference terminology as you read. Most confusion comes from undefined terms.

    3. Skim First, Deep Dive Later¶

    Read document headers and summaries first. Deep dive only when needed.

    4. Follow Cross-References¶

    Specs link to related specs. Follow links when concepts aren't clear.

    5. Use Real Examples¶

    Try building a simple schema as you read. Learning by doing reinforces concepts.

    6. Ask Questions¶

    If documentation is unclear, file an issue. Documentation improves through feedback.
